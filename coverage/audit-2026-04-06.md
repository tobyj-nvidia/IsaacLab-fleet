**Auditor:** Automated coverage audit via Claude Code

## 1. Executive Summary

IsaacLab has **128 test files** containing **~965 test functions** spread across 6 packages. The
`isaaclab` core package has the deepest test suite. However, three critical areas are nearly
untested: the `ui` module (0 tests), the `visualization` module (0 tests), and `isaaclab_tasks`
(391 source files, 19 test functions). Running these tests requires a live NVIDIA Isaac Sim GPU
environment, meaning no tests can be executed in a standard Python environment.

---

## 2. Test Infrastructure

### Framework
- **Test runner:** pytest
- **Configuration:** root `pyproject.toml` under `[tool.pytest.ini_options]`
- **Custom marker:** `isaacsim_ci` — marks tests that require Isaac Sim GPU environment
- **Test discovery:** all `test_*.py` files under `source/*/test/` trees

### Test Execution Requirements
All tests import `isaaclab.app.AppLauncher` and launch Isaac Sim as a prerequisite. This means:
- Tests **cannot run** without a GPU and a licensed Isaac Sim installation (5.1.0 target, 4.5/5.0 compat)
- CI uses self-hosted GPU runners inside Docker containers built from the Isaac Sim base image
- There is no unit-test layer that can run offline

### CI/CD Pipeline (`.github/workflows/`)

| Workflow | Trigger | What Runs |
|---|---|---|
| `build.yml` | PR → devel / main / release/* | Two parallel jobs: `test-isaaclab-tasks` (filter: `isaaclab_tasks`) and `test-general` (filter: `not isaaclab_tasks`) |
| `daily-compatibility.yml` | Scheduled daily 4 AM UTC | Full test suite across Isaac Sim 4.5.0 and 5.0.0 |
| `pre-commit.yaml` | PR + push | Ruff lint/format, codespell, license headers (no test execution) |
| `docs.yaml` | PR + push to main/devel/release | Sphinx documentation build only |
| `license-check.yaml` | PR | License header validation only |

**Test isolation:** Tests run inside Docker containers with full GPU access.
Results are exported as JUnit XML and published as PR comments via `publish-unit-test-result-action`.

## 6. Risk Assessment: Critical Untested Paths

### Critical Risk

| Area | Reason |
|---|---|
| `isaaclab.ui` (10 source files) | **Zero tests.** UI panels for joint controls, recording, etc. — entirely untested. |
| `isaaclab.envs` wrappers & curriculum | 48 source files, 19 tests. Curriculum terms, command generators, observation wrappers — core RL plumbing tested mainly as integration smoke. |
| `isaaclab_tasks` reward functions | 391 source files. No tests verify that reward signals are correctly shaped, clipped, or scaled. A silent reward regression cannot be caught by CI. |
| `isaaclab_mimic` datagen pipeline | The full data-generation graph (subtask detection, annotation, file output) has only 2 smoke tests. Incorrect annotations silently corrupt training data. |

### High Risk

| Area | Reason |
|---|---|
| `isaaclab.devices` | 38 source files (gamepads, XR, retargeters) with 17 tests. Retargeter math and XR device parsing are partially tested but device-specific code paths are not. |
| `isaaclab.scene` InteractiveScene | 3 tests for a class that underpins all environments. Clone handling, articulation registration, and sensor attachment are minimally covered. |
| `isaaclab_rl` wrappers | 4 wrapper integrations (RL Games, SKRL, RSL-RL, SB3) each have 1 smoke test. Observation space reshaping, dict obs, and multi-agent paths are not tested. |
| `isaaclab_assets` config validation | 30 asset configs validated only for "does it load" — no mass, inertia, joint limit, or DoF plausibility checks. |

### Medium Risk

| Area | Reason |
|---|---|
| `isaaclab.terrains` | 17 source files, 2 test files. Complex procedural terrain generators have limited edge-case coverage. |
| `isaaclab.actuators` degradation | PD/DC motor force-saturation, velocity limits, and friction model edge cases not tested. |
| `isaaclab.sensors` edge cases | Good overall coverage but multi-tiled camera and multi-mesh ray caster have only 5 tests each. |

## 7. Recommendations (Prioritized)

### Priority 1 — Fill Critical Gaps

1. **Add `ui` module tests.** Even basic instantiation tests (can widgets initialize without a
   display?) would catch import errors and API regressions. Target: 10–20 tests.

2. **Add reward-shape validation to `isaaclab_tasks`.** For each task, assert that the reward
   is within expected bounds after a fixed number of steps with a scripted policy (e.g.,
   zero-action or random-action). This catches reward scaling bugs without requiring full
   training runs.

3. **Expand `isaaclab.envs` tests.** Curriculum term logic, command sampling, and observation
   normalization should each have dedicated unit-like tests. The `test_manager_based_env.py`
   file has only 2 tests for a 48-file module.

4. **Expand `isaaclab_mimic` datagen tests.** Add tests that verify output HDF5 structure,
   annotation keys, and selection strategy coverage beyond the 2 current tests.

### Priority 2 — Improve Shallow Coverage

5. **Add offline-runnable unit tests.** Many `utils`, `managers`, and `controllers` functions
   are pure Python/PyTorch and do not require Isaac Sim. Extracting these into tests runnable
   without a GPU would dramatically shorten feedback cycles.

6. **Add RL wrapper behavioral tests.** For each RL library wrapper (RSL-RL, SKRL, RL Games,
   SB3), test that observation spaces are correctly reported, actions are correctly clipped,
   and episode resets propagate properly.

7. **Improve `scene` coverage.** Add tests for clone-count variations, articulation root
   offset, and sensor attachment/detachment in `InteractiveScene`.

### Priority 3 — Infrastructure Improvements

8. **Add code coverage reporting.** Integrate `pytest-cov` and publish HTML/XML coverage
   reports as CI artifacts. This would replace estimation with measurement.

9. **Introduce a fast (offline) test tier.** Tag tests that do not need Isaac Sim with a
   `@pytest.mark.offline` marker and run them in a lightweight CI job (no GPU, no Docker).
   This would enable faster PR feedback.

10. **Add mutation testing or property-based tests** for math utilities (`utils/math.py`
    has 38 tests but primarily tests known outputs; fuzz testing transform composition
    would find edge cases).
